# Initial Data Scope

This document defines what we currently want to collect. It is a coverage map, not a fixed schema. New sources and variables may be added; weak sources may be dropped after evidence-based review.

## 1. Betting market data — highest priority

Collect as much timestamped market history as practical across major bookmakers and exchanges.

Candidate markets:

- 1X2 / moneyline
- Asian handicap
- totals / over-under
- both teams to score
- correct score and other stable markets where history is available

For every quote or market state, preserve when available:

- bookmaker / exchange
- market and selection
- line / handicap / total
- price / odds
- opening state
- every available intermediate update
- closing / pre-kickoff state
- update timestamp
- suspension / reopen events
- source-native identifiers
- margin / overround and implied probability (raw and derived kept separately)

Priority is full time-series history, not only open and close.

## 2. Match and team data

- competition, season, round, kickoff time, venue
- home/away teams and canonical IDs
- final and half-time score
- extra time / penalties
- cards, substitutions, major match incidents
- standings and table position at each historical time
- recent form and opponent strength
- home/away splits
- rest days and schedule congestion
- travel and consecutive-away patterns
- cup/league context and match importance proxies
- head-to-head history where useful

Advanced team/match metrics where available:

- xG / xGA
- shots / shots on target
- possession
- corners / set pieces
- pressing / PPDA
- field tilt / territory / progression metrics
- possession value or similar advanced metrics

## 3. Player, lineup, availability, and squad data

- expected lineup where historically timestamped
- confirmed starting XI
- substitutes
- injuries
- suspensions
- late withdrawals
- minutes and workload over rolling windows
- position and role
- national-team duty
- transfers
- goalkeeper changes
- defensive-line combinations
- manager changes

Potential derived variables later:

- lineup continuity
- squad-strength loss
- rotation intensity
- goalkeeper / defense stability
- workload and fatigue indices

## 4. Event-level match data

Preserve raw event streams whenever available rather than only aggregates.

Examples:

- shots and shot attributes
- goals
- passes
- carries / dribbles
- tackles / pressures
- fouls
- cards
- substitutions
- VAR events
- penalties
- event coordinates and timestamps

Raw event data should remain available so future features can be recomputed without recollecting history.

## 5. External context

- weather: temperature, rain, wind, humidity, pressure
- venue and pitch characteristics
- altitude
- neutral venue / closed doors / attendance context
- timezone and travel distance
- referee identity and historical tendencies

## 6. News and text information

Preserve timestamped public information where feasible:

- injury news
- club announcements
- manager press conferences
- player interviews
- transfer news
- trusted reporter updates
- lineup leaks / confirmations
- unusual incidents

Store, when permitted:

- source
- publication timestamp
- original text or source reference
- involved teams / players
- later structured extraction fields
- embeddings or representations generated by later pipelines

## 7. Identity and time fields

The project should converge toward stable canonical identifiers such as:

- `match_id`
- `competition_id`
- `team_id`
- `player_id`
- `bookmaker_id`
- `market_id`
- `source_id`

Critical timestamps include:

- `event_time`
- `known_at`
- source publication/update time
- ingestion time
- timezone / UTC normalization

## 8. Scale philosophy

Do not prematurely restrict collection because a variable seems unimportant today.

Preferred principle:

**Collect broadly, preserve raw, derive later, admit to models selectively.**

Storage cost is secondary to historical recoverability, provenance, timestamp integrity, and source quality.

A large raw lake may contain hundreds of source fields and eventually generate thousands of candidate features. That is acceptable. Model inputs should still be controlled experiment by experiment.
